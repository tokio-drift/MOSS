## 🐛 BUG DETAILS

### BUG #1: Race Condition in pitch.c - pitch_read()
**File**: `src/pitch.c` lines 58-75  
**Severity**: MEDIUM-HIGH  
**Impact**: May deliver wrong balls or skip deliveries  
**Status**: FIXABLE

**Current Code**:
```c
int read_index = (pitch_index - 1 + PITCH_SIZE) % PITCH_SIZE;
ball = pitch_buffer[read_index];
```

**Problem**:
- The circular buffer uses `pitch_index` to track position
- After write: `pitch_index = (pitch_index + 1) % PITCH_SIZE`
- The formula `(pitch_index - 1 + PITCH_SIZE) % PITCH_SIZE` is correct mathematically
- BUT: There's a potential race if writer updates `pitch_index` while reader is computing

**Fix**:
```c
int read_index = pitch_index - 1;
if (read_index < 0) read_index = PITCH_SIZE - 1;
ball = pitch_buffer[read_index];
```

This is simpler and less prone to arithmetic errors.

### BUG #2: Weak Synchronization of innings_over Flag
**File**: `src/main.c` line 24  
**Severity**: MEDIUM  
**Impact**: Program may hang or not terminate innings gracefully  
**Status**: PARTIALLY FIXABLE

**Current Code**:
```c
volatile int innings_over = 0;
```

**Problem**:
- Used in multiple threads WITHOUT mutex protection
- Read in `pitch.c:42`, `pitch.c:61`, `batsman.c`, `bowler.c` without locks
- `volatile` keyword does NOT guarantee thread-safety across cores
- Compiler may still cache the value

**Why It's an Issue**:
```c
// In pitch.c (no lock!):
while (!ball_ready && !innings_over)  // ← Could be cached
    pthread_cond_wait(&ball_consumed_cond, &pitch_mutex);
```

The `innings_over` check OUTSIDE the mutex is problematic.

**Fix Option A** (Best - Use atomic):
```c
#include <stdatomic.h>
_Atomic(int) innings_over = 0;

// Change all reads to:
if (atomic_load(&innings_over)) break;

// Change all writes to:
atomic_store(&innings_over, 1);
```

**Fix Option B** (Current style - Use mutex):
```c
// Wrap all reads:
pthread_mutex_lock(&scheduler_mutex);
int local_innings_over = innings_over;
pthread_mutex_unlock(&scheduler_mutex);
if (local_innings_over) break;
```

### BUG #3: catch_taken Not Reset Between Deliveries
**File**: `src/players/batsman.c` lines 103-113  
**Severity**: MEDIUM  
**Impact**: After first catch, ALL aerial balls are treated as caught  
**Status**: FIXABLE

**Current Code**:
```c
if (r.aerial)
{
    // ... select fielder ...
    notify_fielder(fielder_id, r.aerial);
    
    pthread_mutex_lock(&fielder_mutex);
    player *f = &bowling_team[fielder_id];
    if (!catch_taken && attempt_catch(f, r.aerial))
    {
        catch_taken = true;
        r.wicket = true;
        // ...
    }
    // ← BUG: catch_taken is never set back to false!
}
```

**Problem**:
- `catch_taken` is global and persists across deliveries
- Once ANY fielder takes a catch, the flag stays true
- Next aerial ball will skip the `attempt_catch()` call

**Fix**:
```c
if (r.aerial)
{
    // ... select fielder ...
    notify_fielder(fielder_id, r.aerial);
    
    pthread_mutex_lock(&fielder_mutex);
    player *f = &bowling_team[fielder_id];
    if (!catch_taken && attempt_catch(f, r.aerial))
    {
        catch_taken = true;
        r.wicket = true;
        caught = 1;
        caught_by_keeper = f->is_keeper;
    }
    else  // ← ADD ELSE BLOCK
    {
        catch_taken = false;  // Reset for next ball
        r.wicket = false;
        r.runs = rand() % 3;
    }
    // ← Now catch_taken is reset at the end of each delivery
    pthread_mutex_unlock(&fielder_mutex);
}
```

### BUG #5: Missing Bounds Check in on_wicket()
**File**: `src/scheduler.c` line 137  
**Severity**: MEDIUM  
**Impact**: Rare - potential array out of bounds  
**Status**: FIXABLE

**Current Code**:
```c
int on_wicket()
{
    pthread_mutex_lock(&scheduler_mutex);
    int next = select_next_batsman_locked(batting_team, TEAM_SIZE, &match);
    if (next == -1)
    {
        pthread_mutex_unlock(&scheduler_mutex);
        return -1;
    }
    striker_id = next;  // ← No bounds check if next >= TEAM_SIZE
    next_batsman_id = next + 1;
    pthread_mutex_unlock(&scheduler_mutex);
    return next;
}
```

**Problem**:
- `select_next_batsman_locked()` can return any valid batsman index
- The check `if (next == -1)` is present
- But there's no check `if (next >= TEAM_SIZE)`
- If there's a bug in `select_next_batsman_locked()`, could write out of bounds

**Fix**:
```c
int on_wicket()
{
    pthread_mutex_lock(&scheduler_mutex);
    int next = select_next_batsman_locked(batting_team, TEAM_SIZE, &match);
    if (next == -1 || next >= TEAM_SIZE)  // ← ADD bounds check
    {
        pthread_mutex_unlock(&scheduler_mutex);
        return -1;
    }
    striker_id = next;
    next_batsman_id = next + 1;
    pthread_mutex_unlock(&scheduler_mutex);
    return next;
}
```

## 🏗️ COMPILATION ENVIRONMENT ISSUE

### Problem: Windows MinGW Missing pthread.h

**Error**:
```
gcc: src/main.c:4:21: fatal error: pthread.h: No such file or directory
```

**Cause**:
- Your Windows MinGW installation doesn't have POSIX thread support
- Default MinGW uses Win32 threads, not POSIX pthreads

**Solution A: Reinstall MinGW-w64 with POSIX Threads** (Recommended)
1. Uninstall current MinGW
2. Download MinGW-w64 installer
3. Install with these settings:
   - **Version**: Latest (e.g., 13.x)
   - **Architecture**: x86_64
   - **Threads**: **POSIX** (NOT Win32)
   - **Exception Handling**: dwarf2 or seh
4. Add to PATH
5. Try compilation again

**Solution B: Use WSL (Windows Subsystem for Linux)**
1. Install WSL2
2. Install Ubuntu 22.04
3. In WSL terminal:
   ```bash
   sudo apt update
   sudo apt install build-essential
   cd /mnt/c/Users/parsh/OS-T20
   make clean && make
   ```

**Solution C: Use Online Linux Environment**
